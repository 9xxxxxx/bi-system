# M2 收口后 M3-M6 多 Agent 并行开发计划

## 0. 当前基线

本计划基于 2026-07-17 已验收的 M2 基线继续推进。M0 工程基础、M1 数据接入和 M2 数据建模均已完成；M2 验收证据见 `docs/verification/m2-verification.md`，迁移到新机器的交接说明见 `docs/plans/2026-07-17-development-handoff.md`。

M3-M6 尚未进入生产代码实现。下一步必须先完成 M3-R0 架构与技术验证，冻结图表配置、布局、筛选、模板和查询合同，再进入迁移、API 或前端页面开发。

## 1. 目标

在 M2 数据建模完成验收后，以最多 4 个并发 Agent 推进 M3-M6。阶段之间保持依赖顺序，阶段内部按后端、前端和质量边界并行开发，主 Agent 负责合同冻结、集成、提交和验收。

本计划不改变 V1 范围。各阶段业务边界以需求规格为准；任何新增能力、数据库专属实现或外部服务接入必须先更新架构决策和阶段计划。

## 2. 启动条件：M2 收口闸门

以下条件全部满足后，才允许启动 M3 生产代码开发：

- 公共指标、计算字段、行级策略、资源权限和受控查询形成可运行闭环。
- 查询入口能够统一执行工作区隔离、资源权限、行级权限、版本固定、超时和结果规模限制。
- M2 前端完成关系模型、数据集、字段、指标、权限和查询预览的目标范围。
- SQLite 与 PostgreSQL 的迁移升降级、核心查询语义和权限结果一致。
- 常规查询 P95、PostgreSQL 20 并发、桌面与 390 px 移动视口通过验收。
- 后端、前端、pre-commit 和 PostgreSQL 全量检查通过。
- `docs/verification/m2-verification.md` 已提交，工作区干净，M2 接口和迁移已形成稳定基线。
- `docs/plans/2026-07-17-development-handoff.md` 已提交，新机器可通过 Git 取得当前开发状态和继续开发入口。

M2 收口期间可以并行编写 M3-M6 的详细计划、ADR 和无生产依赖的技术验证，但不得提前提交依赖未冻结 M2 合同的业务实现。

## 3. 并行开发模型

### 3.1 Agent 职责

每轮最多同时运行 4 个 Agent，包括主 Agent：

| 角色 | 主要职责 | 默认文件边界 |
| --- | --- | --- |
| 主 Agent / 集成负责人 | 拆分任务、冻结合同、处理共享文件、代码审查、集成提交、阶段验收 | 路由汇总、应用壳、共享合同、迁移链、配置、CI、计划和验收记录 |
| Agent A / 后端领域 | 领域模型、服务、API、后台任务、数据库可移植性 | `backend/src/bi_system/` 对应功能包、后端测试 |
| Agent B / 前端体验 | 页面、编辑器、交互状态、响应式布局、前端 API 接入 | `frontend/src/features/` 对应功能包、组件测试 |
| Agent C / 质量与平台 | 跨数据库、性能、安全、浏览器、导出或算法专项 | 集成测试、基准脚本、专项适配器、验证证据 |

角色按任务轮次调整。专业性较强的导出、编辑器和 AI 任务可以临时替换 Agent C 的职责，但主 Agent 始终保留集成所有权。

### 3.2 共享工作区规则

- 主 Agent 在每轮开始前发布任务清单、文件所有权和禁止修改列表。
- 两个 Agent 不得同时编辑同一文件；共享类型、路由汇总、迁移头和应用壳由主 Agent 统一修改。
- Agent 开始前必须检查 `git status --short`，将已有未提交改动视为其他开发者所有。
- 子 Agent 不执行 `git add`、`git commit`、迁移重排或依赖升级，除非主 Agent在任务中明确授权。
- Agent 完成后报告修改文件、合同变化、验证命令、未解决风险，不自行扩大范围。
- 主 Agent 按任务串行审查和提交，提交信息遵循 Conventional Commits，确保每个提交可独立回滚。
- 发生合同冲突、迁移冲突或同文件重叠时暂停相关 Agent，由主 Agent先固定基线再重新派发。

### 3.3 每轮工作流

1. 主 Agent确认上一轮工作区干净、测试通过并冻结本轮输入合同。
2. 将任务拆成最多三个互不重叠的子任务，同时明确接口、文件边界和验收命令。
3. 子 Agent并行实现并运行局部测试；主 Agent同步处理共享合同或准备集成测试。
4. 主 Agent逐项审查、合并共享改动、运行受影响测试并形成聚焦提交。
5. 本轮完成后执行后端静态检查、前端检查和必要的双数据库测试，再进入下一轮。
6. 阶段末执行完整验收并提交验证记录；未通过阶段闸门不得启动下一阶段生产代码。

## 4. M3：BI 核心

### 4.1 交付目标

交付从已激活数据集创建仪表盘的基础闭环：仪表盘与模板、桌面和移动布局、核心图表、字段配置、筛选、预览、保存和权限控制。

M3 不实现跨组件联动、下钻、穿透、参数、书签、规则告警和通报编辑，这些能力分别留给 M4 和 M5。

### 4.2 并行轮次

#### M3-R0：架构与技术验证

| Agent | 任务 | 输出 |
| --- | --- | --- |
| 主 Agent | 固定仪表盘、页面、组件、布局、模板和查询合同 | M3 详细计划、API 合同、状态机和 ADR |
| Agent A | 验证图表配置到 M2 查询 AST 的服务端编译边界 | 可运行 spike、错误矩阵、性能结论 |
| Agent B | 验证图表库、拖拽布局和移动端只读布局 | 可运行 spike、桌面/移动截图、包体积结论 |
| Agent C | 设计标准数据集、图表快照和 M3 验收矩阵 | 测试数据、基准方法、验收清单 |

只有通过技术验证的库才能进入生产依赖；版本、许可证和选择依据写入架构评估。

#### M3-R1：领域基础与工作台骨架

| Agent | 任务 | 主要交付 |
| --- | --- | --- |
| Agent A | 仪表盘、版本、页面、组件、模板和布局模型 | 双数据库迁移、领域服务、约束测试 |
| Agent B | 仪表盘列表、编辑器壳、组件面板、属性面板 | 懒加载路由、加载/空/错误状态、组件测试 |
| Agent C | 图表查询合同、配置校验和测试夹具 | 强类型合同、非法配置测试、标准数据集 |

主 Agent负责迁移链、路由注册和前后端共享合同集成。

#### M3-R2：核心图表与筛选

| Agent | 任务 | 主要交付 |
| --- | --- | --- |
| Agent A | 图表查询服务、权限注入、查询结果序列化 | 查询 API、超时/截断、结构化错误 |
| Agent B | KPI、表格、柱状、条形、折线、面积、饼/环图 | 字段槽、聚合、排序、Top N、图例和标签 |
| Agent C | 全局/页面/组件筛选及日期筛选合同 | 筛选合并规则、权限回归、跨数据库测试 |

#### M3-R3：布局、模板与验收

| Agent | 任务 | 主要交付 |
| --- | --- | --- |
| Agent A | 模板实例化、版本保存、回收站和引用保护 | 生命周期 API、并发版本冲突处理 |
| Agent B | 拖拽缩放、复制粘贴、桌面/移动独立布局 | 编辑体验、只读移动视图、响应式测试 |
| Agent C | 性能、浏览器、权限和双数据库验收 | P95、截图、回归测试和缺陷清单 |

### 4.3 M3 完成闸门

- 从空白画布和模板均可创建、保存、重新打开仪表盘。
- 核心图表计算结果与 M2 查询结果及人工核算一致。
- 全局、页面和组件筛选具有确定的合并顺序并继承行级权限。
- 桌面可编辑，移动端可查看和筛选，布局互不覆盖。
- 普通页面、缓存首屏和常规查询达到需求规格中的性能目标。
- SQLite/PostgreSQL、前端检查、浏览器验收和 `m3-verification.md` 全部通过。

## 5. M4：高级分析

### 5.1 交付目标

在稳定的 M3 仪表盘模型上增加联动、层级下钻与上钻、明细穿透、参数、个人视图、书签、高级图表配置、规则告警和查询性能优化。

### 5.2 并行轮次

#### M4-R0：交互上下文合同

- 主 Agent：冻结筛选上下文、数据批次、指标版本、层级路径和权限传播规则。
- Agent A：设计服务端交互查询与明细穿透合同。
- Agent B：设计前端交互状态机、面包屑和跨组件事件模型。
- Agent C：构建权限泄漏、循环联动和高基数压力测试。

#### M4-R1：联动、下钻与穿透

- Agent A：实现联动查询、层级解析、受控明细查询和审计上下文。
- Agent B：实现点击过滤、下钻/上钻、面包屑和明细视图。
- Agent C：验证所有交互继承工作区、行级权限、版本与批次上下文。

#### M4-R2：参数、书签与高级配置

- Agent A：实现参数合同、个人视图和书签持久化。
- Agent B：实现参数控件、书签管理、双轴、条件格式、目标线和参考区间。
- Agent C：扩展瀑布、散点、热力、漏斗、矩形树图等标准图表兼容测试。

#### M4-R3：告警与性能

- Agent A：规则告警、调度状态和系统内通知事件。
- Agent B：告警配置、状态反馈和性能感知交互。
- Agent C：查询缓存、索引建议、并发、超时和常规联动 P95 验收。

### 5.3 M4 完成闸门

- 联动、下钻、穿透、参数和书签均可恢复且不会绕过权限。
- 高级图表配置序列化后可稳定重放，不依赖任意代码。
- 常规联动 P95 小于 2 秒，复杂查询显示进度并受超时限制。
- 告警具有可追溯规则版本、数据批次、触发结果和通知状态。
- 完成双数据库、桌面/移动浏览器和 `m4-verification.md` 验收。

## 6. M5：通报系统

### 6.1 交付目标

交付结构化通报编辑、模板、单人编辑锁、自动保存、版本历史、不可变发布快照、撤回、修订、导出和系统内通知。

### 6.2 并行轮次

#### M5-R0：编辑器与导出验证

- 主 Agent：冻结内容块、模板变量、快照、发布和修订状态机。
- Agent A：验证编辑锁、自动保存冲突和不可变快照模型。
- Agent B：验证结构化编辑器、中文粘贴、分页和锁定块。
- Agent C：验证 PDF、Word、PNG 和打印的跨平台生成链路。

#### M5-R1：通报领域与编辑器基础

- Agent A：通报、模板、内容块、版本、编辑锁和迁移。
- Agent B：结构化块编辑器、目录、模板变量和自动保存界面。
- Agent C：锁超时、接管、冲突恢复和版本一致性测试。

#### M5-R2：图表引用与发布快照

- Agent A：解析 M3/M4 图表和指标引用，生成固定版本快照。

#### M5-R3：导出与通知

- Agent A：后台导出任务、状态、产物存储、权限和审计事件。

### 6.3 M5 完成闸门

- 单人锁、自动保存、冲突恢复、管理员接管和超时释放可验证。

- 发布快照固定数据批次、指标版本、筛选条件和图表配置。
- PDF、Word、PNG 和打印结果经过视觉检查，表格可导出 Excel/CSV。

## 7. M6：智能与加固

### 7.1 交付目标

在统一查询与权限入口上增加自然语言问数、自动摘要与图表、趋势预测、异常检测、贡献度分析、数据脱敏网关，并完成审计、备份恢复、安全和 Windows/Linux 跨平台验收。

### 7.2 并行轮次

#### M6-R0：安全与评估基线

- 主 Agent：冻结数据分级、模型调用、人工确认、证据展示和失败策略。
- Agent A：设计只读智能查询编排，禁止模型直连数据库。
- Agent B：设计对话、证据、确认、保存和失败状态体验。
- Agent C：建立脱敏、越权、提示注入、幻觉和评估数据集。

#### M6-R1：脱敏网关与自然语言问数

- Agent A：权限校验、指标解析、查询计划验证、脱敏和模型适配器。
- Agent B：问数对话、查询确认、口径与筛选证据、图表保存流程。
- Agent C：敏感/严格敏感数据策略、结果规模、复杂度和攻击测试。

#### M6-R2：预测、异常和贡献分析

- Agent A：采用经过验证的算法库实现预测、异常和贡献度服务。
- Agent B：结果解释、置信信息、失败原因和主动保存界面。
- Agent C：离线评估、基线对比、误报漏报和性能测试。

#### M6-R3：生产加固

- Agent A：审计覆盖、会话安全、备份恢复和后台任务可观测性。
- Agent B：审计检索、系统管理、安全确认和无障碍回归。
- Agent C：Windows/Linux、SQLite/PostgreSQL、20 用户并发、恢复演练和安全验收。

### 7.3 M6 完成闸门

- 模型不能直接访问数据库，所有查询复用 M2-M4 的权限与查询入口。
- 敏感数据调用前确认，严格敏感数据禁止问数，发送内容经过可验证脱敏。
- 智能结果展示数据集、指标口径、筛选、生成时间、证据和失败原因。
- AI 生成内容只能由用户主动保存，不能自动发布。
- 审计默认保留 180 天且不包含令牌、密码或未脱敏数据。
- 完成备份恢复、跨平台、性能、安全和 `m6-verification.md`，V1 方可宣布完成。

## 8. 全程质量门禁

每个集成提交至少执行受影响测试；每轮结束执行对应模块检查；每个里程碑结束执行完整门禁：

```powershell
uv sync --locked --all-groups
uv run ruff check backend scripts
uv run ruff format --check backend scripts
uv run basedpyright backend/src backend/tests scripts
uv run pytest backend/tests -q --cov=bi_system
uv run python scripts/run_postgres_tests.py
npm --prefix frontend ci
npm --prefix frontend run check
npm --prefix frontend run build
uv run pre-commit run --all-files
```

涉及可见界面时必须补充桌面和 390 px 移动视口截图；涉及编辑器、图表、导出或 3D/Canvas 等渲染表面时必须执行浏览器像素和非空检查。涉及迁移、共享查询、权限、导出和 AI 时必须同时验证 SQLite 与 PostgreSQL，并添加越权和跨工作区负向测试。

## 9. Git 与交付规范

- 每个任务形成聚焦、可独立审查的 Conventional Commit。
- 迁移、后端合同、前端接入和验收记录分开提交，避免混合大提交。
- 依赖变更必须同步 `uv.lock` 或 `frontend/package-lock.json`，并记录许可证和选择依据。
- 主 Agent提交前运行 `git diff --check`，确认没有暂存其他 Agent 或用户的无关改动。
- 阶段验收提交使用 `docs: record Mx verification evidence`。
- 未通过当前里程碑验收时，不创建下一阶段生产功能提交。

建议提交序列如下：

```text
docs: define M3 dashboard architecture
feat(db): add dashboard domain foundation
feat(api): add dashboard management
feat(web): add dashboard workspace
feat(analytics): add dashboard filters and queries
docs: record M3 verification evidence
```

M4-M6 使用相同原则按领域、API、前端、专项能力和验收拆分，不预先锁定不准确的提交数量。

## 10. 风险与停止条件

出现以下任一情况时暂停并行开发，由主 Agent先解决基线问题：

- M2 或当前阶段共享合同仍在频繁变化。
- 两个 Agent 需要编辑同一核心文件或同一迁移。
- SQLite 与 PostgreSQL 行为不一致且尚无适配策略。
- 新依赖未完成许可证、维护状态、包体积或跨平台验证。
- 权限、行级策略、快照或脱敏存在绕过路径。
- 阶段测试持续失败，无法区分基线缺陷和新功能缺陷。
- 工作区存在来源不明或与当前任务重叠的未提交改动。

暂停不等于取消阶段。主 Agent应缩小任务边界、恢复稳定合同和绿色测试后，再重新派发互不重叠的工作。

## 11. V1 完成定义

只有 M0-M6 的实现、验证记录和需求规格第 13 节验收标准全部满足，才可宣布 V1 完成。完成状态必须能够追溯到迁移、自动化测试、性能数据、浏览器截图、导出样例、安全测试和跨平台恢复记录，不能仅以功能页面存在或单数据库测试通过作为依据。
